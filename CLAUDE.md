# CLAUDE.md

このリポジトリでClaudeが作業する際のガイドラインです。

## リポジトリの目的

組込みソフトウェア開発部門に配属される新入社員向けの講義資料（60分）を管理する。

## ファイル構成

```
.
├── slides.md          # Marpスライド本体
├── speaker_notes.md   # スピーカーノート（話す内容・問いかけ・流れ）
├── README.md          # リポジトリの説明
└── CLAUDE.md          # このファイル
```

## スライドの設計方針

### 腹落ちさせるための構造

スライドはキーワード・図・箇条書きに絞る。「なぜ」の説明は講師が口頭で話す。
話す内容の流れは「痛み → なぜ → 解決策」。

- 痛み：現場でこういうことが起きる、という具体的な困りごと
- なぜ：その痛みが起きる根本的な理由（腹落ちの核心）
- 解決策：だからこの概念・ツール・プロセスが存在する

### たとえの方針

機械系出身者（プログラミング経験が限定的）が「あ、それと同じだ」と思えるたとえを使う。

- ウォーターフォール → 図面を確定させてから製造に入る
- V字モデル → 図面を描いた人が検査基準を作る
- Git → 設変管理台帳が自動化されたもの
- プロセスとSOP → 誰がやっても同じ品質が出せる

### スライドに載せてよいもの

- キーワード・用語
- 箇条書き（1行1概念）
- 比較表
- コードブロック（図・例示）
- 一言まとめ（blockquote）

### スライドに載せないもの

- 段落文章（読む文章にしない）
- 「なぜ」の詳細説明（speaker_notes.mdに書く）
- 具体的なエピソード（speaker_notes.mdに書く）

## speaker_notes.mdの構成

各スライドに対応するセクションを持つ。各セクションに以下を記載する。

- 話す流れ（痛み → なぜ → 解決策の展開）
- 具体的なエピソード・たとえ
- 問いかけ（該当する箇所のみ）

## 対象受講者

- 機械系学部卒の新入社員
- プログラミング経験は限定的
- ソフトウェア開発の知識はほぼゼロからのスタート

## 修正時の注意

- スライドを修正したらspeaker_notes.mdの対応箇所も確認する
- 技術的な正確さを保つ（特にA-SPICE・V字モデル・Verification/Validationの説明）
- 「腹落ちするか」を常に基準にする。用語の羅列にしない
